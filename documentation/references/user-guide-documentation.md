# User Guide Documentation

Use this reference for user guides, admin guides, onboarding, tutorials,
how-to guides, troubleshooting, FAQ, support docs, and public integration
narratives.

## Required Sources

- Product UI, screenshots or generated assets, support issues, analytics,
  release notes, API examples, admin settings, and existing help content.
- Diataxis for document type separation: https://diataxis.fr/

## Default Paths

- `docs/user-guides/user-guide.md`
- `docs/user-guides/admin-guide.md`
- `docs/user-guides/onboarding-guide.md`
- `docs/user-guides/troubleshooting-faq.md`
- `docs/user-guides/support.md`
- `docs/api/integration-guide.md`

## Required Content

- Audience and task.
- Prerequisites, permissions, and setup.
- Task steps with exact UI labels, commands, or API calls.
- Expected results.
- Troubleshooting or support path.
- Safety, privacy, or data-handling warnings when user action has consequences.

## Workflow

1. Choose the Diataxis type before drafting: tutorial, how-to, reference, or
   explanation.
2. Keep tutorials controlled and beginner-safe.
3. Keep how-to guides task-specific and action-oriented.
4. Keep reference pages complete and lookup-friendly.
5. Keep explanations focused on why the system behaves as it does.
6. Public docs must be sanitized and must not expose internal scoring,
   infrastructure, security internals, or confidential data.

## Validation

- Steps can be followed in order.
- UI labels and API examples match the product.
- Public docs have no internal-only detail.
- Support paths and prerequisites are current.
