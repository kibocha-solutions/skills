# **Documentation Architecture and Agent-Skill Design**

## **Executive Conclusion**

The synthesis of authoritative technical writing frameworks, engineering paradigms, and JetBrains Writerside mechanical constraints reveals that an optimal documentation architecture relies on semantic, decoupled structures rather than filesystem-bound hierarchies. The traditional practice of using numeric folder prefixes is an anti-pattern. While local consistency can justify exceptions, the default rule is to avoid numeric prefixes because Writerside tree files control navigation order and topic filenames affect IDs/URLs. This convention is generally avoided to prevent friction during refactoring, URL generation, and search engine optimization.  
A modern documentation ecosystem is best conceptualized as a "Documentation Library." This model serves as a master repository composed of isolated, audience-specific "Instances." These instances leverage independent XML-based tree files to construct their tables of contents dynamically. This architecture allows a single repository to generate distinct public, internal, and restricted portals without duplicating content or exposing confidential engineering doctrine to unauthorized external users. Furthermore, it strictly adheres to the cognitive separation principles of the Diátaxis framework, ensuring that tutorials, how-to guides, explanations, and reference materials remain philosophically distinct and mechanically segregated.  
Regarding agent-skill boundaries, creating separate artificial intelligence skills for different document types—such as distinct agents for API documentation, architecture decision records, or operational runbooks—introduces severe skill sprawl and context bloat. Because the generation of these documents shares the same underlying natural language execution model, linguistic style guidelines, and validation pipelines, they must be governed by a single, \*\*core routing\*\* "documentation" skill. This master skill acts as a routing intelligence, deploying lean, domain-specific templates and reference payloads into the context window precisely when triggered by the user's intent. The sole exception is the "technical-diagrams" skill, which necessitates strict architectural isolation due to its fundamentally distinct execution engine, its reliance on mxGraph XML source generation, and its requirement for visual rendering pipelines. Ultimately, the architecture must align entirely with Writerside's paradigm: semantic file naming, tree-driven navigation, structured landing pages, and extensive reuse through XML snippets.

## **Documentation Architecture Models Compared**

The vocabulary used to describe a collection of technical documentation profoundly influences how authors structure it and how users consume it. The term "book" implies a linear, sequential reading path, which directly contradicts the non-linear, search-driven consumption patterns of modern software engineering documentation. A "knowledge base" typically implies an unstructured, organically grown wiki environment lacking strict governance. A "portal" refers strictly to the published web interface rather than the underlying repository structure.  
The most accurate and effective mental model is a "Documentation Library." A library suggests a curated, overarching collection of distinct, purposeful volumes that share a common indexing system and governance standard but serve vastly different audiences and operational needs.19  
Within this library model, the actual information architecture relies on synthesizing three major organizational frameworks: Diátaxis, Red Hat Modular Documentation, and DITA. Diátaxis organizes documentation exclusively around the user's cognitive state and immediate operational needs, rigidly dividing content into four quadrants. Tutorials are learning-oriented and require a controlled environment; How-to Guides are task-oriented and address real-world problems; Reference materials are information-oriented and focus on accuracy and completeness; Explanations are understanding-oriented and provide historical or theoretical context. This framework excels at preventing document overlap, ensuring that a step-by-step procedure does not become bloated with theoretical background logic.  
Red Hat Modular Documentation focuses heavily on reusability and docs-as-code efficiency.6 It defines an "Assembly" as a curated collection of modules that accomplish a specific user story. The modules themselves are strictly typed as Concept, Procedure, or Reference. Concept modules explain what something is, Procedure modules provide numbered, imperative steps, and Reference modules offer static lookup data.6 This aligns closely with DITA, which similarly categorizes topics into Concept, Task, Reference, and Troubleshooting paradigms.5  
The structural comparison indicates that the modular approaches of Red Hat and DITA map perfectly to Writerside's mechanical features. A Writerside "Instance" serves as the top-level publication environment. A Writerside "Section starting page" serves as the entry point to a Red Hat Assembly, organizing the sub-modules into a coherent user story.4 Writerside "Topics" map to individual Concept, Procedure, or Reference modules, which can be reused across different instances using semantic XML inclusion tags.3 Diátaxis provides the qualitative standard for writing the prose within these modules, ensuring that the tone and structure match the module's declared cognitive purpose. The documentation successfully avoids becoming a flat, unnavigable dictionary of disconnected files by utilizing Writerside's tree files to enforce a rigid, hierarchical table of contents that guides the reader through predefined, logical learning paths.1

## **Writerside Structure Findings**

The mechanical constraints and features of JetBrains Writerside dictate precisely how the documentation library must be structured at the code level. The fundamental unit of publication in Writerside is the "Instance." An instance is an isolated output configuration defined by a .tree file containing hierarchical \<toc-element\> tags.1 Each instance possesses its own table of contents, a strictly designated home page, distinct versioning metadata, and a specific web-path for deployment.20 Multiple instances must be leveraged when a single repository is directed at distinct audiences. For example, a repository should compile one instance for public API consumers and a completely separate instance for internal system maintainers, drawing from the same pool of source files but presenting entirely different navigation trees.2  
Writerside's Table of Contents is deliberately decoupled from the physical filesystem directory structure.1 Authors manage the navigation hierarchy exclusively within the XML tree file. This decoupling introduces powerful architectural features that must be utilized. For instance, "Empty TOC elements" are used to group topics logically in the navigation sidebar without forcing the author to create a superficial introductory topic page.1 When a user clicks an empty TOC element, the navigation tree merely expands to reveal child topics; it does not load a new URL or alter the main reading pane.  
Conversely, when a visual overview or catalogue is necessary, Writerside provides the \<section-starting-page\> semantic markup.4 A starting page is a highly specialized XML topic that acts as a landing dashboard for a section. These pages are optional and should only be used when a section truly needs orientation or a visual catalogue. It organizes child topics into interactive grids, cards, and spotlight sections using hierarchical tags such as \<spotlight\>, \<primary\>, \<secondary\>, and \<misc\>.4 These pages provide critical reading paths and intuitive "start here" functionality for users navigating complex architectural or operational hierarchies. Because they rely on strict XML schemas to generate the user interface elements, starting pages cannot be authored in standard Markdown.4  
Another crucial feature of the Writerside paradigm is the "Hidden topic," activated by applying the hidden="true" attribute to a \<toc-element\> within the tree file.1 Hidden topics are compiled into the final web output and remain fully indexed for search engines and internal search bars, but they are entirely omitted from the visible navigation sidebar.1 This mechanism is highly recommended for sprawling reference indexes, highly localized troubleshooting targets, or exhaustive architectural decision records that are only relevant when explicitly searched for, thereby preventing the navigation interface from becoming overwhelmed by peripheral files.  
Content reuse in Writerside must be managed strategically without overengineering the repository. The optimal approach involves creating a dedicated "Library Instance" configured with the is-library="true" attribute.2 A library instance does not generate standalone web output; instead, it serves purely as a centralized repository for \<snippet\> elements.3 These snippets can be dynamically injected into any public or internal topic using the \<include\> tag. This ensures that shared components, such as authentication prerequisites, environment variable definitions, or critical security warnings, are maintained in a single source of truth and propagated accurately across all published instances.3 When files are renamed or removed, Writerside supports graceful deprecation via the redirection-rules.xml file, allowing authors to use the accepts-web-file-names attribute to seamlessly route legacy URLs to new semantic targets, preventing 404 errors.21

## **Naming Standards and Verdict on Numeric Prefixes**

The use of numeric prefixes on directories and files—such as 00-project-control, 01-architecture, or 02-security—is a legacy convention stemming from early filesystems and rudimentary static site generators that relied exclusively on alphabetical sorting to dictate navigation order.22 In the context of a modern docs-as-code repository, and specifically within the Writerside ecosystem, numeric prefixes are classified as a severe anti-pattern and are strongly discouraged.  
When numeric prefixes are applied to folders and files, they inevitably leak into the generated URLs.23 A URL containing a numeric prefix, such as https://docs.example.com/01-architecture/04-data-model.html, inflicts multiple layers of damage on the documentation ecosystem. It harms Search Engine Optimization by inserting irrelevant characters into the slug, reduces human readability, and creates immense friction during long-term maintenance. If an author needs to insert a new architectural topic between 01 and 02, they are forced either to renumber all subsequent files—which breaks all existing inbound links and generates massive, noisy Git diffs—or to introduce decimal hacks like 01a, which destroys the aesthetic and structural integrity of the library.22 Because Writerside explicitly uses the .tree file to control the logical order of the table of contents, embedding ordering metadata directly into the physical filesystem is entirely redundant and actively fights the compiler's intended design.1  
The recommended standard follows the Google Developer Documentation Style Guide, which mandates strictly lowercase, hyphenated names (kebab-case) for all directories and files, utilizing only standard ASCII alphanumeric characters.7 Folders must be named semantically based on their domain, product area, or logical grouping, yielding clean directories such as architecture/, security/, and api/. Topic filenames must reflect the exact content precisely, resulting in files like data-classification-policy.md rather than generic or numbered placeholders.7  
Writerside relies on the physical filename to automatically generate the id of the topic, which subsequently dictates the generated web-file-name.20 By default, Writerside replaces special characters with hyphens and converts all uppercase letters to lowercase.20 Providing clean, lowercase-hyphenated filenames natively ensures that the filesystem path, the topic ID, and the resulting URL slug are perfectly aligned without requiring manual overrides. The navigation titles actually displayed to the user in the sidebar are controlled entirely separately via the toc-title attribute in the tree file or the title declared inside the topic markup itself.1 This ensures complete separation of concerns between underlying filesystem mechanics and user-facing presentation.

## **Documentation Library Structure**

The overarching mental model is a Documentation Library, centralized within a single Git repository but logically partitioned to serve multiple distinct audiences through isolated Writerside instances. The physical filesystem must prioritize semantic categorization over output grouping, as individual markdown topics may be included in multiple instances simultaneously. The repository root must contain the Writerside configuration directories alongside a primary documentation directory housing the semantic domain folders.20  
The core directory structure must follow semantic boundaries rather than chronological execution order. The root configuration folder manages the writerside.cfg file, which registers all instances, defines global variables, and establishes web paths.20 A dedicated snippets directory acts as the central hub for reusable content, keeping boilerplate code, standard security warnings, and shared prerequisites isolated from the main narrative flow.3  
Within the primary documentation folder, domains are strictly segregated. The architecture directory houses system context narratives, overarching domain models, and the chronological log of architectural decision records. The API directory contains the raw OpenAPI specifications, REST reference pages, and GraphQL schemas. The security directory isolates threat models, data classification matrices, and baseline configuration standards. The operations directory is dedicated to functional execution, housing deployment pipelines, infrastructure as code documentation, and strictly formatted incident response runbooks. Testing, guides, and reference directories similarly isolate their respective cognitive types. Finally, a releases directory manages the chronological changelogs, migration notes, and semantic versioning records.  
Within the Writerside interface, logical structuring takes absolute precedence over physical location. "Section starting pages" should be utilized as catalogue pages for major domains where a section needs an entry point or catalogue.4 For example, the operations directory should feature a landing page that utilizes visual cards to link to specific, high-priority runbooks, drastically reducing the cognitive load for engineers responding to an active incident. Deep reference material, such as deprecated architectural decisions or exhaustive database schema definitions, must exist as topics hidden from the table of contents using the hidden="true" attribute. This prevents sidebar bloat while ensuring the content remains accessible via search or direct hyperlinks from higher-level catalogue pages.1

## **Public, Internal, Restricted, and Confidential Separation**

Separating audiences is critical to ensure that public users receive sanitized, polished guidance while internal developers retain access to raw architectural decisions, and security reviewers can view highly confidential threat models without risking exposure. This separation must be executed dynamically using Writerside Instances rather than maintaining separate repositories, which inevitably leads to documentation drift.2  
A single library repository should define multiple .tree files to represent these audience boundaries. The public instance tree compiles the external-facing portal, including integration guides, public API references, and sanitized release notes. The internal engineering instance compiles a comprehensive internal portal, including everything in the public documentation plus internal architectural records, operational runbooks, and testing strategies. Finally, the restricted security instance compiles the compliance portal, featuring raw threat models, precise infrastructure topologies, and specific confidential decisions that cannot be exposed broadly.  
By structuring the library through instances, public documentation cannot inadvertently leak private internal doctrine. The continuous integration pipeline can be configured to build these instances as entirely separate artifacts, deploying the public instance to an open content delivery network while heavily restricting the internal and security instances behind zero-trust corporate authentication proxies.  
Note: Instance separation is a necessary but insufficient safeguard. Each reusable topic must carry an explicit sensitivity classification (e.g., \`classification: public\`) in its frontmatter or metadata. This allows CI/build rules to enforce separation. Classification sensitivity flows upward: A lower-sensitivity topic (e.g., Public) may be reused in a higher-sensitivity instance (e.g., Internal or Restricted). A higher-sensitivity topic (e.g., Restricted or Confidential) must never be reused in a lower-sensitivity instance (e.g., Public) unless it is explicitly sanitized and reclassified. For Msngi, public documentation must never draw from private scoring logic, security internals, or donor/beneficiary-sensitive material.

## **Mandatory vs. Conditional vs. Good-to-Have Documentation**

Implementing a rigid, universal mandate that demands every document type for every project creates catastrophic overhead and developer fatigue. Documentation must be classified based on specific architectural, temporal, and operational triggering conditions.  
**Mandatory Early Documentation:** These records are strictly non-negotiable for any software project moving beyond a simple prototype script. They establish the foundational constraints of the system. The *README* acts as the primary entry point. The *Project Brief* and *Scope Boundaries* are required immediately to prevent scope creep. An *Architecture Overview* utilizing a C4 Context diagram is mandatory to establish what external systems are being interacted with. Finally, *Architecture Decision Records (ADRs)* are crucial early on to track foundational technology choices—such as database selection or framework choice—before codebase inertia sets in and makes those decisions irreversible.25  
**Trigger-Based Mandatory Documentation:** These documents remain entirely optional until specific architectural, operational, or business conditions are met, at which point they become strictly mandatory compliance artifacts. A *Security Threat Model* and *Data Classification matrix* are triggered the exact moment the application handles Personally Identifiable Information (PII), implements user authentication, accepts file uploads, or processes financial transactions.14 *REST or GraphQL API Documentation* is triggered only when an API is exposed to external third parties or distinct, decoupled front-end teams. *Operations Runbooks* and *Monitoring Documentation* are triggered immediately prior to the first production deployment, specifically when an on-call rotation is formally established.16 *Migration Notes* are triggered only when a breaking change to a database schema or a public API is successfully merged.  
**Good-to-Have Documentation:**  
These documents provide high value but can be deferred without blocking production releases. *Contribution Guidelines* are highly recommended for open-source initiatives but represent optional overhead for small, tight-knit internal teams. *Tutorials* are immensely valuable for public developer platforms and SDKs, but drafting them for internal CRUD applications is typically an overinvestment. Code Comment Guides are valuable because while linters can enforce docstring \*presence\* or \*formatting\*, they cannot reliably enforce whether a comment explains \*contract\*, \*intent\*, \*invariants\*, \*side effects\*, or \*authorization assumptions\*. The detailed reference guide remains necessary for quality.

To prevent documentation paralysis, a stage-based prioritization model dictates exactly what needs to be written at each lifecycle phase, ensuring engineering velocity is maintained without sacrificing critical knowledge capture.

## **Skill vs. Reference vs. Template Decision Matrix**

The core meta-research task requires mapping these complex documentation patterns into the architecture of an agent skills repository. The governing decision rule formulated for this task is: *A separate agent skill is justified only when the task demands a fundamentally distinct execution model, recurring tool use, a strict external validation pipeline, or possesses enough complexity that keeping it inside a general documentation skill would shatter the agent's context window.*  
Based on this strict rule, the widespread proliferation of individual skills for different document types must be categorically rejected. Creating separate agent skills for "API documentation," "Security documentation," and "Architecture Decision Records" causes severe skill sprawl. Large language models tasked with natural language generation utilize identical underlying execution mechanics regardless of whether they are generating a markdown SRE runbook, a markdown ADR, or a markdown user guide. Forcing the orchestration layer to choose between twenty different documentation skills introduces massive routing hallucinations.

| Candidate Pattern | Decision | Architectural Rationale |
| :---- | :---- | :---- |
| **General Documentation** | **Keep as Core Skill** | Acts as the core router. Governs prose generation, formatting, Writerside mechanics, and template injection.1  |
| **Technical Diagrams** | **Separate Skill** | Strictly requires a distinct mxGraph/SVG XML execution engine, palette asset awareness, complex geometric calculations, and visual QA pipelines. Fundamentally hostile to standard prose generation. |
| **Architecture / ADRs** | **Template/Reference** | ADRs are simply markdown files adhering to the MADR Y-statement template. The LLM relies entirely on standard prose generation.30 |
| **Security Documentation** | **Template/Reference** | Threat modeling and data classification rely on applying OWASP checklists to markdown templates.26 No unique execution engine required. |
| **SRE / Runbooks** | **Template/Reference** | Runbooks require strict structural formatting (Symptoms, Diagnosis, Remediation) easily enforced via a markdown template injected dynamically by the general skill.28 |
| **API Documentation** | **Defer / Merge** | API workflows vary widely (code-first, contract-first, manually maintained specs). The skill must be flexible, documenting API outputs from the available source of truth (OpenAPI file, code routes, DRF serializers, GraphQL schema, etc.).  |
| **Changelogs** | **Reference** | Simply requires adherence to "Keep a Changelog" formatting rules during generation.18 |

The agent requires only two overarching skills: documentation and technical-diagrams. The documentation skill acts as an intelligent, context-aware router. When a user requests an architecture decision record, the documentation skill parses the intent, retrieves the MADR template from its internal reference library, injects it into the context window alongside the Google Style Guide rules, and executes the prose generation seamlessly.

## **Recommended Final Documentation Skill Reference Structure**

To support the agent in generating highly specialized, technically accurate documentation without relying on separate skills, the documentation skill repository must contain a robust, hierarchical reference library. These references provide the targeted context injections the agent needs to perform specialized tasks accurately.  
The references/ directory must contain the overarching rules of the system. A writerside-mechanics.md file dictates the rules for XML tags, hidden topics, section starting pages, and tree file management.1 A diataxis-framework.md file provides cognitive guidance on adjusting tone, structure, and verb usage depending on whether the output is a Tutorial or Reference material. A google-microsoft-style.md file enforces instructions for kebab-case file naming, active voice implementation, and the elimination of biased or militaristic terminology.7  
The templates/ directory provides the structural scaffolds. The madr-adr-template.md provides the strict MADR template requiring Context, Options, Decision, and Consequences.25 The sre-runbook-template.md enforces the critical Symptoms, Diagnosis, Remediation, and Verification lifecycle.28 The owasp-threat-model.md provides the framework for establishing trust boundaries and mitigating data flow vulnerabilities.14 The keep-a-changelog.md file contains the exact formatting rules for semantic versioning releases 18, and a writerside-starting-page.xml file offers the precise XML boilerplate required to generate interactive grids and cards.4 When the agent executes the documentation skill, it utilizes semantic routing to fetch the appropriate combination of reference rules and templates, merging them with the user's raw input to generate the final documentation perfectly aligned with the target standard.

## **Recommended Final Technical-Diagrams Cross-Routing**

Because the technical-diagrams skill must remain strictly isolated due to its complex mxGraph execution model, the two agent skills must possess explicit, reliable cross-routing protocols to handle hybrid requests.  
When a user requests a comprehensive Architecture Overview, the orchestration layer triggers the documentation skill. The agent drafts the high-level prose explaining the system's external dependencies. When the agent determines a diagram is necessary, the documentation skill explicitly \*\*invokes\*\* the technical-diagrams skill. The diagram skill generates the Draw.io XML source and exports the rendered SVG asset into the target repository. Once completed, the orchestration layer returns the asset path to the documentation skill, which seamlessly replaces the placeholder with the embedded image link before finalizing the file.  
Conversely, if a user directly asks the technical-diagrams skill to architect a Threat Model diagram, the skill generates the visual topology identifying the trust boundaries. It subsequently routes a request back to the documentation skill, passing the generated topology as context, instructing it to generate the accompanying OWASP text analysis based on the diagram's identified vulnerabilities.

## **Recommended Writerside Instance, Tree, and Topic Naming Pattern**

To guarantee cohesion across the Writerside build pipeline and prevent URL degradation, naming protocols must be strictly enforced by the agent.  
Topics, whether authored in Markdown or semantic XML, must utilize lowercase-hyphenated semantic names, such as api-authentication.md. Writerside uses this physical filename to seamlessly construct the internal topic ID and the resulting web file name.20 Under no circumstances may numeric prefixes like 01-api.md be used, as they severely corrupt the URL structure.23 Tree files must reflect the instance output audience clearly, resulting in files named public-docs.tree or internal-dev.tree.2  
For table of contents grouping, folder structures on the disk must not dictate the navigation order. Authors and agents must directly edit the \<toc-element\> tags within the .tree file to establish hierarchical relationships and the intended cognitive reading order.1 Finally, starting pages must be named intuitively, such as operations-overview.topic, and placed at the root of their semantic folder, utilizing the \<section-starting-page\> XML markup to generate navigational landing zones.4

## **Migration Plan for the Existing Source Canons**

If existing source canons or legacy documentation repositories utilize deprecated naming conventions, particularly numeric prefixes like 01-architecture or 02-security, they must undergo a rigorous, structured refactoring process to align with the new architectural standards.  
The first phase requires flattening and renaming. The agent must systematically remove all numeric prefixes from every directory and file name, renaming all assets to standard kebab-case. Because Writerside requires maintaining inbound links to prevent catastrophic 404 errors for existing users, the second phase involves generating redirection rules. The agent must populate the redirection-rules.xml file, mapping the legacy web file names to the new semantic targets, and apply the accepts-web-file-names attribute directly within the tree file.21  
The third phase requires constructing the tree. The agent maps the old filesystem-based order into the new .tree file using nested \<toc-element\> tags. If a legacy directory previously relied on a basic index.md file for navigation, the agent converts that file into a robust Writerside \<section-starting-page\> to generate modern card interfaces. Finally, execution isolation requires moving all raw drawing files out of the documentation prose directories and into a dedicated assets repository governed exclusively by the technical-diagrams skill.

## **Risks and Anti-Patterns**

Implementing a large-scale documentation library carries several severe architectural risks that must be actively mitigated. The most common anti-pattern is filesystem coupling—the assumption that the operating system folder structure should mimic the visual navigation structure. In Writerside, the .tree file must be the sole arbiter of navigation; relying on folders creates rigid, unmaintainable hierarchies.1  
Skill sprawl is a significant operational risk. Attempting to create an autonomous LLM agent for every single document type fractures context windows and needlessly replicates basic prose-generation logic across multiple siloed skills, destroying the agent's routing accuracy. Premature optimization is a project management risk, where teams mandate heavy documentation artifacts—like enterprise Threat Models or exhaustive SRE Runbooks—during the early coding stage, inevitably leading to immediate staleness and intense developer fatigue.  
Table of contents bloat destroys usability. Putting every single file into the visible sidebar overwhelms the reader. Reference dictionaries, old changelogs, and legacy ADRs must utilize Writerside's hidden="true" attribute so they remain highly searchable without dominating the visual hierarchy.1 Finally, relying on Git commit logs to serve as a changelog is a toxic anti-pattern. Commit logs are inherently noisy, granular, and machine-focused; proper changelogs must follow the "Keep a Changelog" standard and be meticulously curated for human comprehension.17

## **Open Questions**

While the current architecture comprehensively resolves the primary information architecture and agent skill boundaries, several advanced operational questions remain for future refinement. It must be determined how the system should automatically deprecate or flag Architecture Decision Records when the underlying codebase fundamentally shifts, preventing architectural drift. While Open-API specifications are currently deferred as reference artifacts, it remains to be seen whether Writerside can natively parse and render external Swagger JSON dynamically, or if it will require the documentation skill to convert them into Markdown artifacts prior to compilation. Finally, diagram synchronization remains a complex challenge: if an underlying C4 architectural model is updated via infrastructure-as-code, the exact continuous integration trigger required to force the technical-diagrams skill to regenerate the SVG and notify the documentation skill of the update must be formally established.
